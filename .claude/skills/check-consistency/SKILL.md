# 設計書間 整合性チェック

`docs/` 配下の設計書間の整合性をチェックポイント単位でチェックする skill。

**方針:**
- 横断的なチェックポイントごとに `Explore` サブエージェントを並列起動する。
- サブエージェントの報告には事実誤認が含まれる可能性があるため、メインエージェントが再読して検証してから最終レポートを出す。
- 報告のみ。修正はユーザーの判断に委ねる。

## 引数

```
$ARGUMENTS
```

- 引数にファイルパスまたはコンポーネント名を空白区切りで指定すると、対象を絞る（例: `watcher.md api.md`、`watcher dispatcher`）
- 引数なしの場合は下記「対象ファイル」セクションの全設計書を対象とする

## 対象ファイル

- `docs/architecture/*.md`
- `docs/*.md`（ただし `docs/migration/` 以下は除外 — バージョン固有の記録であり相互整合性の対象外）
- `CLAUDE.md`（設計書ではないが、プロジェクト内リンク・参照整合性のチェック対象に含める）

英語版 (`docs_en/`) は対象外（翻訳品質は `translate-docs` skill の管轄）。

## チェックポイント

### A. 参照整合性
- 相対リンクの参照先ファイルが実在するか
- セクションアンカー (`#xxx`) が参照先ファイルの見出しと一致するか
- 削除・リネームされたドキュメントへの参照が残っていないか

### B. 数値・定数の一致
複数ドキュメントで重複記述されている以下が一致しているか:
- 環境変数のデフォルト値・制限値・タイムアウト・リトライ回数
- K8s リソースの request / limit
- SQL カラムの型・長さ
- URL パス・ポート番号
- 単位換算（秒 / ミリ秒、MiB / GiB 等）の違いも検出する

### C. スキーマの重複記述
- DB カラム定義（名前・型・制約）が `database.md` と他ドキュメント（`api.md` / `cjobctl.md` / `watcher.md` 等）で一致しているか
- API フィールド名・型が `api.md` と `cli.md` / `cjobctl.md` で一致しているか
- K8s ラベル名（`cjob.io/*` 等）が `dispatcher.md` / `watcher.md` / `kueue.md` で一致しているか
- K8s リソース種別・annotations の表記が一致しているか

### D. 状態遷移・ライフサイクルの一致
- ジョブ状態（`QUEUED` / `HELD` / `DISPATCHING` / `DISPATCHED` / `RUNNING` / `SUCCEEDED` / `FAILED` / `CANCELLED` / `DELETING` 等）の集合が複数ドキュメントで同一か
- 状態遷移グラフ（どの状態からどの状態に遷移できるか）が複数ドキュメントで矛盾していないか
- 各状態の意味・遷移条件の記述が複数ドキュメントで一致しているか

### E. 用語・命名のゆれ
- 同一概念の表記ゆれ（例: `Dispatcher` / `ディスパッチャ`、`Watcher` / `Reconciler`）
- コンポーネント名・ツール名の統一（`CJob` / `cjob`、`cjobctl` の表記）
- 略語の使用一貫性（初出箇所での定義有無）
- 同義語が混在していないか（例: 「ジョブ投入」「ジョブ送信」「ジョブ submit」）

### F. データフロー・責務の矛盾
- ドキュメント A で「X コンポーネントが Y を書き込む」と記載されていて、ドキュメント B で別のコンポーネントが同じテーブル / リソースを書き込むと記載されていないか
- コンポーネント間の依存方向が矛盾していないか
- sequence diagram 的記述（「Dispatcher が K8s Job を作成 → Watcher が RUNNING 遷移を検出」等）がドキュメント間で一貫しているか
- 責務の粒度がドキュメントごとに異なる場合があるため、「言及の有無」は不整合とみなさず、**明確な矛盾のみ**を報告する

### G. 古い記述の残存
- 最近変更された設計書で、同じ概念を扱う他のドキュメントが古いままになっていないか
- 削除・非推奨化された機能への言及が残っていないか
- `git log --since=...` または最近のコミットで変更された設計書を起点に、関連ドキュメントが追随しているかをチェック

## 手順

### Step 1: スコープ決定

1. 引数を解析して対象ファイルを決定する
   - 引数なし → 上記「対象ファイル」のすべてを対象
   - ファイル名指定あり → 指定ファイル + それらが言及される他のドキュメント
   - コンポーネント名指定あり → `check-drift` のマッピング表に基づき、該当する設計書群 + 関連ドキュメント
2. 対象ファイルのリストを確定させる

### Step 2: チェックポイントごとの並列委譲

7 つのチェックポイント (A〜G) それぞれに対して `Agent` ツールで `Explore` サブエージェントを起動する。**7 つすべてを 1 つのメッセージ内で並列に呼ぶ**こと（複数 `Agent` tool use を単一メッセージに含める）。

各サブエージェントに渡すプロンプトには以下を含める:

1. チェックポイントの説明（上記 A〜G の該当項目を抜粋）
2. 対象ファイルリスト
3. 出力形式（下記「サブエージェントの出力形式」）
4. thoroughness は `very thorough` を指定する

### サブエージェントの出力形式

```
## チェックポイント <A/B/C/...>: <名前>

### [CRITICAL/WARN/INFO] 不整合の内容（1 行要約）
- 発生箇所:
  - `docs/architecture/watcher.md:69` — `該当記述の短い抜粋`
  - `docs/architecture/dispatcher.md:42` — `該当記述の短い抜粋`
- 詳細: 何がどう食い違っているか
- 推奨対応: どのファイルに合わせるべきか、または要確認ポイント

... （不整合ごとに繰り返し）

問題が見つからなかった場合は「問題なし」と明記する。
```

severity:
- `CRITICAL` — 読者に誤解を与える、または設計の矛盾を示唆する不整合（状態遷移の矛盾、スキーマ定義の食い違い、責務の矛盾 等）
- `WARN` — 事実関係は矛盾していないが、数値の食い違い・リンク切れ・古い記述など整備すべきもの
- `INFO` — 用語ゆれや軽微な表現の違いで、実質的な読解を妨げないもの

### Step 3: 報告の検証（必須）

サブエージェントの報告には事実誤認が含まれる可能性があるため、**最終レポートに含める前に必ず以下を実施する**:

1. 各 finding について、引用されているすべての該当箇所（`<file>:<line>`）を `Read` ツールで直接再読する
2. 本当に不整合があることを自分の目で確認する
   - サブエージェントが文脈を見落として誤検出している可能性がある（例: 同じ単語でも別の意味で使われていた）
   - 数値の比較で単位換算を間違えている可能性がある
   - 状態遷移の比較で、文脈が違う（正常系 vs 異常系など）のを取り違えている可能性がある
3. 確認できた finding のみを最終レポートに含める
4. 誤検出だった finding は破棄する（ユーザーには提示しない）
5. 検証時に追加の不整合に気付いた場合は、独立した finding として追加する
6. 検証で破棄した finding の件数も最終レポートの先頭サマリに記載する（透明性のため）

### Step 4: 最終レポート

検証済みの finding をユーザーに提示する。以下を含める:

- **サマリ**: 総 finding 数、severity 別内訳、検証で破棄した件数、対象ファイル数・チェックポイント数
- **チェックポイント別詳細**: 各 finding の severity・1 行要約・該当箇所・不整合の詳細・推奨対応
- **対応方針の補足**: 各不整合について、どのドキュメントを正とすべきかの推定は提示するが、最終判断はユーザーに委ねる

## 注意事項

- ドキュメントは日本語で書かれているため、用語ゆれの検出では同義語・表記ゆれを両方考慮する
- 数値の重複記述チェックでは、単位（秒 / ミリ秒、MiB / GiB 等）の違いを必ず正規化して比較する
- F. データフロー矛盾では、責務の記述粒度がドキュメントごとに異なる場合があるため、明確な矛盾のみ報告する（粒度の違い・言及の有無は不整合ではない）
- 状態名のゆれ（大文字小文字、日本語化の有無）は E で、状態集合・遷移条件の違いは D で扱う
- チェックポイント間で重複検出が起きた場合（例: 同じ不整合が C と E の両方で検出された）、最終レポートでは 1 件にまとめてチェックポイントを併記する
