# ユーザーガイド

ジョブの状態（QUEUED, RUNNING, SUCCEEDED など）の一覧と遷移については [README のジョブの状態](../README.md#ジョブの状態)を参照してください。

## 1. ジョブ数の制限

CJob では、システムの安定性と公平性を保つために、ユーザーごとにジョブ数の上限が設けられています。

| 制限の内容 | 上限 | 上限に達したときの動作 |
|---|---|---|
| 投入できるジョブ数 | 500 件 | 新しいジョブを投入できなくなる |
| 同時に実行できるジョブ数 | flavor ごとに 32 件、合計 50 件 | 超過分は待機状態で順番を待つ |

### 1.1 投入できるジョブ数の上限（500 件）

`cjob add` や `cjob sweep` で投入できるジョブの数には上限があります。待機中のジョブとキャンセル済みのジョブの合計が **500 件** を超えると、新しいジョブを投入できなくなります。実行中のジョブはカウントに含まれないため、実行中のジョブが投入枠を圧迫することはありません。キャンセルしただけではジョブ数の枠は空きません。

上限に達した場合は、`cjob delete` でキャンセル済みのジョブを削除してください。不要な待機中のジョブがあれば、`cjob cancel` でキャンセルした上で削除することでも枠を空けられます。ジョブの削除について詳しくは[完了済みジョブの削除](../README.md#完了済みジョブの削除)を参照してください。

### 1.2 同時に実行できるジョブ数の上限（flavor ごとに 32 件、合計 50 件）

一度に実行状態になれるジョブは、**flavor ごとに最大 32 件** です。たとえば CPU ジョブと GPU ジョブはそれぞれ独立して 32 件まで同時に実行できます。CPU ジョブが 32 件実行中でも、GPU ジョブの枠には影響しません。

さらに、すべての flavor を合わせた **合計のジョブ数にも上限（デフォルト 50 件）** があります。たとえば CPU ジョブが 30 件、GPU ジョブが 20 件で合計 50 件に達した場合、どちらの flavor でも新しいジョブは実行されず待機状態になります。

33 件目以降のジョブや合計上限に達した場合は、待機状態（QUEUED）のまま順番を待ち、実行中のジョブが完了すると自動的に次のジョブが開始されます。

基本的には投入した順番に実行されますが、リソースの空き状況によっては後から投入した小さいジョブが先に実行される場合があります。

> [!NOTE]
> 完了したジョブは内部的に一定時間（デフォルト設定では 5 分間）残存します。この間も合計のジョブ数としてカウントされるため、短時間に多くのジョブが完了する場合、実際に実行状態になれるジョブ数は上限よりも少なくなることがあります。残存期間が過ぎると自動的に枠が解放されます。`cjob usage` の Resource Quota セクションで現在のジョブ数を確認できます。

## 2. sweep を使った効率的なジョブ実行

パラメータを変えながら同じプログラムを何度も実行したい場合は、`cjob sweep` を使うと便利です。

`cjob sweep` は実行回数にかかわらず **ジョブ 1 件分** としてカウントされます。たとえば `-n 100` で 100 回実行しても、投入できるジョブ数の上限（500 件）に対しては 1 件しか消費しません。同時に実行できるジョブ数（32 件）についても同様です。大量の実行を行いたい場合は、`cjob add` で 1 件ずつ投入するよりも `cjob sweep` を使うほうが上限を気にせず済みます。

### 2.1 基本的な使い方

たとえば、パラメータ `--trial` の値を 0 から 99 まで変えて 100 回実行したい場合を考えます。

`cjob add` で 1 件ずつ投入すると 100 回コマンドを実行する必要がありますが、`cjob sweep` なら 1 回のコマンドで済みます。

```bash
# 100 回分のジョブを一度に投入し、10 件ずつ並列で実行する
cjob sweep -n 100 --parallel 10 -- python main.py --trial _INDEX_
```

- `-n 100`: 全部で 100 回実行する
- `--parallel 10`: 同時に 10 件まで並列で実行する
- `_INDEX_`: 各実行に自動で 0, 1, 2, ... , 99 の番号が割り当てられる

### 2.2 `_INDEX_` の活用方法

`_INDEX_` は実行ごとに異なる番号（0 から始まる連番）に自動で置き換えられます。この番号を使ってパラメータを変えることができます。

```bash
cjob sweep -n 50 --parallel 5 -- python train.py --seed _INDEX_
```

スクリプトファイルの中では `$CJOB_INDEX` という変数名で番号を直接参照できます。

```bash
# run.sh の中身の例
python train.py --config configs/config_${CJOB_INDEX}.yaml
```

```bash
cjob sweep -n 10 --parallel 5 -- bash run.sh
```

### 2.3 並列数の選び方

`--parallel` は 1 つの sweep ジョブの中で同時に実行するタスク数です。並列数を増やすとその分だけ CPU やメモリを同時に消費するため、クラスタのリソース状況に応じて適切な値を設定してください。

```bash
# 10 タスクずつ並列で実行
cjob sweep -n 100 --parallel 10 -- python main.py --trial _INDEX_

# 並列数を控えめにしてリソースの消費を抑える
cjob sweep -n 100 --parallel 5 -- python main.py --trial _INDEX_
```

### 2.4 タスクごとのリソース指定

`--cpu`、`--memory`、`--gpu` オプションを使うと、**1 つのタスクが使う計算リソースの量** を指定できます。sweep 全体ではなく、並列で動く個々のタスクそれぞれに割り当てられる量です。

```bash
# 各タスクに CPU 2 コア、メモリ 4GiB を割り当てて実行する
cjob sweep -n 50 --parallel 5 --cpu 2 --memory 4Gi -- python main.py --trial _INDEX_

# GPU を使う場合は --gpu で 1 タスクあたりの GPU 数を指定する
cjob sweep -n 10 --parallel 2 --gpu 1 -- python train.py --trial _INDEX_
```

- `--cpu`: 1 タスクが使う CPU コア数（省略時: 1）
- `--memory`: 1 タスクが使うメモリ量（省略時: 1Gi）
- `--gpu`: 1 タスクが使う GPU 数（省略時: 0、つまり GPU を使わない）

たとえば `--parallel 5 --cpu 2 --memory 4Gi` と指定した場合、最大で 5 タスクが同時に動くため、合計で CPU 10 コア・メモリ 20GiB を同時に消費することになります。クラスタのリソースに余裕がない場合はタスクが待機状態になるため、並列数やリソース量は必要に応じて調整してください。

### 2.5 実行時間の上限について

`--time-limit` で設定する実行時間の上限は、**すべてのタスクが終わるまでの合計時間** に対してかかります。1 回ごとの実行時間ではありません。

たとえば `-n 100 --parallel 10 --time-limit 2h` と指定した場合、100 回分のタスクが 2 時間以内にすべて完了する必要があります。2 時間を過ぎると、まだ残っているタスクも含めてジョブ全体が中止されます。

タスク数が多い場合や並列数が少ない場合は、その分だけ完了までの時間が長くなります。十分な余裕を持った値を設定してください。

### 2.6 sweep のログ確認

sweep で実行した各ジョブのログは、番号を指定して個別に確認できます。

```bash
# 全タスクのログをまとめて表示
cjob logs 3

# 番号 5 のタスクのログだけ表示
cjob logs 3 --index 5

# 番号 5 のタスクのログをリアルタイムで追跡
cjob logs --follow 3 --index 5
```

### 2.7 sweep のキャンセル

sweep ジョブをキャンセルすると、進行中の全タスクがまとめて中止されます。特定の番号だけを中止することはできません。

```bash
cjob cancel 3
```

## 3. 計算リソースの種類を指定する（`--flavor`）

クラスタには CPU だけのノードや GPU を搭載したノードなど、性能の異なる複数の種類の計算ノードがあります。ジョブを投入するときに `--flavor` オプションを使うと、どの種類のノードで実行するかを指定できます。

### 3.1 基本的な使い方

```bash
# 通常の CPU ノードで実行する（--flavor を省略した場合のデフォルト）
cjob add -- python main.py

# GPU ノードで実行する
cjob add --flavor gpu --gpu 1 -- python train.py --epochs 100
```

`--flavor` を省略すると、管理者が設定したデフォルトの種類（通常は CPU ノード）が自動的に選ばれます。

### 3.2 利用可能な種類の確認

`cjob flavor list` で利用できるノードの種類を確認できます。`*` が付いている種類がデフォルト（`--flavor` を省略した場合に使われる種類）です。

```
$ cjob flavor list
NAME             GPU    NODES    DEFAULT
cpu              -      2          *
gpu              yes    1
```

各種類のリソースの上限を確認するには `cjob flavor info` を使います。

```
$ cjob flavor info gpu
name:   gpu
GPU:    対応

RESOURCE        QUOTA   TASK LIMIT
CPU                32           32
Memory          128Gi        128Gi
GPU                 4            4
```

- **QUOTA**: その種類のノード全体で使えるリソースの合計
- **TASK LIMIT**: 1 つのジョブ（タスク）に指定できるリソースの上限

### 3.3 sweep での使い方

`cjob sweep` でも同様に `--flavor` を指定できます。

```bash
# GPU ノードで 10 回分のタスクを 2 件ずつ並列で実行する
cjob sweep -n 10 --parallel 2 --flavor gpu --gpu 1 -- python train.py --trial _INDEX_
```

### 3.4 指定した種類のリソースに合わせる

`--flavor` で指定した種類のノードが持つリソースの範囲内でジョブを投入してください。たとえば、GPU を持たない種類のノードに対して `--gpu 1` を指定するとエラーになります。

```
$ cjob add --flavor cpu --gpu 1 -- python train.py
Error: flavor 'cpu' は GPU をサポートしていません
```

CPU やメモリの要求量が `cjob flavor info` で表示される **TASK LIMIT** を超える場合もエラーになります。

### 3.5 ジョブの種類を確認する

投入したジョブがどの種類のノードで実行されているかは `cjob status` で確認できます。

```
$ cjob status 1
job_id:        1
status:        RUNNING
command:       python train.py --epochs 100
cwd:           /home/jovyan/project-a
flavor:        gpu
cpu:           1
memory:        1Gi
gpu:           1
time_limit:    24h (残り 23h 50m)
...
```

## 4. ジョブの実行を保留する（`cjob hold` / `cjob release`）

投入したジョブの実行を一時的に保留し、あとからキューに戻すことができます。たとえば、システムのメンテナンスが予告されているときに、ジョブが実行中に強制停止されるのを避けるために使えます。

### 4.1 基本的な使い方

```bash
# ジョブ 5 の実行を保留する
cjob hold 5

# 保留を解除してキューに戻す
cjob release 5
```

保留中のジョブは `cjob list` で `HELD` ステータスとして表示されます。保留中はシステムによる実行が行われません。`cjob release` でキューに戻すと、通常の待機ジョブと同様に順番が来たら自動的に実行されます。

### 4.2 複数のジョブをまとめて保留する

`cjob cancel` や `cjob delete` と同様に、範囲指定や複数指定ができます。

```bash
# 範囲指定
cjob hold 1-10

# 複数指定
cjob hold 1,3,5

# 組み合わせ
cjob hold 1-5,8,10-12

# 待機中（QUEUED）のジョブを全て保留する
cjob hold --all
```

解除も同様です。

```bash
# 保留中（HELD）のジョブを全てキューに戻す
cjob release --all
```

### 4.3 実行時間の上限で絞り込んで保留する

`cjob list` の `--time-limit` オプションと `--format ids` オプションを組み合わせると、実行時間の上限（`--time-limit` で設定した値）が一定以上のジョブだけを絞り込んで、まとめて保留できます。

たとえば、メンテナンスまで 6 時間しかないときに、6 時間以上かかるジョブだけを保留するには次のようにします。

```bash
# 待機中で実行時間の上限が 6 時間以上のジョブの ID 一覧を確認
cjob list --status QUEUED --time-limit 6h: --format ids

# 確認した結果を使ってまとめて保留
cjob hold $(cjob list --status QUEUED --time-limit 6h: --format ids)
```

`--time-limit` は `<下限>:<上限>` の形式で範囲を指定します。下限は「以上」、上限は「未満」です。

```bash
# 実行時間の上限が 6 時間以上のジョブ
cjob list --time-limit 6h:

# 実行時間の上限が 12 時間未満のジョブ
cjob list --time-limit :12h

# 実行時間の上限が 6 時間以上 12 時間未満のジョブ
cjob list --time-limit 6h:12h
```

`--format ids` はジョブの ID をコンマ区切りで出力します。`cjob hold` や `cjob cancel` など、ID を受け取るコマンドにそのまま渡せます。

### 4.4 保留できるジョブの条件

保留できるのは **QUEUED（待機中）** のジョブだけです。すでに実行中（RUNNING）やディスパッチ処理中（DISPATCHING / DISPATCHED）のジョブは保留できません。

保留中のジョブはキャンセルすることもできます。

```bash
# 保留中のジョブをキャンセル
cjob cancel 5
```

## 5. プログラムが使う CPU コア数と `--cpu` の合わせ方

`--cpu` はシステムがジョブに割り当てる CPU コア数です。一方、プログラムが実際に何コア使うかは、プログラム自身の設定によって決まります。**この 2 つが合っていないと、リソースを効率的に使えません。**

たとえば `--cpu 4` を指定したのに、プログラムが 1 コアしか使わない設定になっていると、残りの 3 コア分が無駄になります。逆に `--cpu 1` のまま、プログラムが多数のコアを使おうとすると、割り当てられた範囲を超えて処理が遅くなる場合があります。

### 5.1 プログラムが複数コアを使うケース

プログラムが複数の CPU コアを使うケースは、大きく分けて 2 つあります。

**ライブラリが自動的に複数コアを使うケース**

NumPy・SciPy などの数値計算ライブラリの多くは、行列演算などの処理を高速化するために、内部で自動的に複数のコアを使います。この場合、利用者がコードの中で並列処理を書いていなくても、複数コアが使われます。

**自分で並列処理を書くケース**

Python の `concurrent.futures` や `multiprocessing` などを使って、自分で複数のプロセスやスレッドを立ち上げる場合も、その分だけ CPU コアを消費します。たとえば `ProcessPoolExecutor(max_workers=4)` と書いた場合、最大 4 コアを同時に使います。

どちらの場合も、プログラムが使うコア数の合計が `--cpu` で指定した値と合うように調整してください。

### 5.2 プログラム側の使用コア数を調整する

プログラムが使用するコア数の調整方法は、プログラムやライブラリによって異なります。ジョブを投入する前に、使用するプログラムやライブラリのドキュメントを確認してください。

よくある例として、数値計算ライブラリでは環境変数でコア数を制御できる場合があります。

| 環境変数 | 対象の例 |
|---|---|
| `OMP_NUM_THREADS` | OpenMP を使うプログラム全般 |
| `OPENBLAS_NUM_THREADS` | OpenBLAS（NumPy 等が内部で使用することがある） |
| `MKL_NUM_THREADS` | Intel MKL（NumPy 等が内部で使用することがある） |

これらはあくまで一例です。使用するプログラムによっては、別の環境変数やコマンドライン引数、設定ファイルでコア数を指定する場合もあります。

自分で並列処理を書く場合は、コードの中でプロセス数やスレッド数を `--cpu` の値に合わせてください。

### 5.3 設定例

`--cpu` で指定したコア数に合わせて、プログラム側の使用コア数も設定してください。

```bash
# コマンドの先頭で環境変数を指定する方法
cjob add --cpu 4 -- OMP_NUM_THREADS=4 python main.py

# cjob sweep の場合
cjob sweep -n 50 --parallel 5 --cpu 4 -- \
  OMP_NUM_THREADS=4 python main.py --trial _INDEX_
```

`export` で事前に設定しておく方法もあります。CJob はジョブ投入時に export 済みの環境変数を引き継ぐため、以降のジョブすべてに同じ設定が適用されます。

```bash
export OMP_NUM_THREADS=4

cjob add --cpu 4 -- python main.py
cjob sweep -n 50 --parallel 5 --cpu 4 -- python main.py --trial _INDEX_
```

## 6. リソース使用状況の確認（`cjob usage`）

`cjob usage` を実行すると、現在のリソース消費状況と直近の使用実績を確認できます。

### 6.1 Resource Quota（現在のリソース消費）

現在使用中のリソース量と、割り当てられた上限が表示されます。

```
Resource Quota
──────────────────────────────────────────────────
  Resource       Used       Hard  Remaining    Use%
  CPU           280.0      300.0       20.0   93.3%
  Memory        800Gi     1250Gi      450Gi   64.0%
  GPU               1          4          3   25.0%
  Jobs             10         50         40   20.0%
```

- **Used**: 現在の使用量
- **Hard**: 割り当てられた上限
- **Remaining**: 残り（`Hard - Used`）
- **Use%**: 使用率

CPU・Memory・GPU は実行中のジョブが要求しているリソースの合計です。Jobs はシステム内部に存在するジョブの数で、[§1.2 の合計のジョブ数の上限](#12-同時に実行できるジョブ数の上限flavor-ごとに-32-件合計-50-件)に対応します。

### 6.2 Resource Usage（日別の使用実績）

直近 7 日間の日別リソース消費量が表示されます。

```
Resource Usage (past 7 days)
──────────────────────────────────────────────────
  Date              CPU (core·h)    Mem (GiB·h)
  2026-03-23               24.0           48.0
  2026-03-24               12.5           25.0
  2026-03-25                8.0           16.0
  ────────────────────────────────────────────────
  Total                    44.5           89.0
```

- CPU は core·h（コア時間）、メモリは GiB·h（ギビバイト時間）の単位で表示されます
- GPU を使用している場合は GPU 列（GPU·h）も表示されます
- リソース消費量はジョブが実行状態になった時点で、`--time-limit` の値をもとに計上されます

## 7. ユーザー設定の管理（`cjob config`）

`cjob config` コマンドを使って、CJob の動作をカスタマイズできます。設定は `~/.config/cjob/config.toml` に保存されます（環境変数 `XDG_CONFIG_HOME` が設定されている場合は `$XDG_CONFIG_HOME/cjob/config.toml`）。

### 7.1 環境変数の除外設定

CJob はジョブ投入時にシェルの環境変数をすべて引き継ぎますが、引き継ぎたくない変数がある場合は除外リストに追加できます。

```bash
# 除外する環境変数を追加
cjob config add env exclude MY_SECRET_TOKEN
cjob config add env exclude AWS_SECRET_ACCESS_KEY

# 除外リストから削除
cjob config remove env exclude MY_SECRET_TOKEN
```

除外リストに追加した環境変数は、`cjob add` や `cjob sweep` でジョブを投入する際にサーバーへ送信されなくなります。

### 7.2 現在の設定を確認する

```bash
cjob config list
```

```
[env]
exclude = [
    "MY_SECRET_TOKEN",
    "AWS_SECRET_ACCESS_KEY",
]
```

設定ファイルが存在しない場合や何も設定していない場合は、デフォルト値（空のリスト）が表示されます。
